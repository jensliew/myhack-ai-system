# Requirements Document

## Introduction

Nexora is a production-ready, full-stack AI-powered ecosystem relationship intelligence platform. It automates mentor-startup matching, verification workflows, and engagement tracking for innovation ecosystems. The platform serves three roles (Admin, Startup, Mentor) with role-based dashboards, integrates real Gemini API for AI recommendations, and stores all interaction data in Firestore for behavioral learning. The UI is a clean, professional SaaS design in light mode only.

## Glossary

- **Platform**: The Nexora web application built with Next.js App Router, TypeScript, Tailwind CSS, shadcn/ui, and Zustand
- **Auth_System**: The Firebase Authentication module handling user registration, login, session management, and role assignment
- **Router**: The Next.js App Router responsible for route protection and role-based redirection
- **Startup_Dashboard**: The dashboard view presented to users with the startup role
- **Mentor_Dashboard**: The dashboard view presented to users with the mentor role
- **Admin_Dashboard**: The dashboard view presented to users with the admin role
- **AI_Engine**: The AI recommendation module integrating Gemini API with Gemma as offline fallback
- **Verification_Engine**: The AI module that extracts, summarizes, and recommends approval decisions for applications
- **Firestore**: The Firebase Firestore database storing all platform data
- **Storage_Service**: The Firebase Storage service handling file uploads and access control
- **Matching_Service**: The service coordinating mentor-startup matching workflows including AI suggestions and interest expressions
- **Document_Service**: The service managing document uploads, visibility, and retrieval
- **Startup_Card**: A UI component displaying startup information in the mentor discovery view
- **Mentor_Card**: A UI component displaying mentor information in the startup matching view
- **Compatibility_Score**: A numerical score (0-100) generated by the AI_Engine representing mentor-startup fit
- **Interest_Record**: A Firestore document recording a mentor's expressed interest in a specific startup
- **Relationship_Record**: A Firestore document representing an accepted mentor-startup pairing

## Requirements

### Requirement 1: User Registration with Role Selection

**User Story:** As a new user, I want to register and select my role (startup or mentor) during signup, so that I am directed to the appropriate dashboard.

#### Acceptance Criteria

1. WHEN a user submits the registration form with valid email, password, and role selection (startup or mentor), THE Auth_System SHALL create a Firebase Authentication account and a corresponding user document in Firestore with the selected role.
2. WHEN a user completes registration, THE Router SHALL redirect the user to the dashboard corresponding to their selected role.
3. IF a user submits a registration form with an email already in use, THEN THE Auth_System SHALL display an error message indicating the email is already registered.
4. IF a user submits a registration form with a password shorter than 8 characters, THEN THE Auth_System SHALL display a validation error requesting a stronger password.
5. THE Auth_System SHALL require users to provide a startup ID or mentor ID as their primary entity identifier during registration.

### Requirement 2: User Login and Session Management

**User Story:** As a registered user, I want to log in and have my session persist, so that I can access my dashboard without re-authenticating on every visit.

#### Acceptance Criteria

1. WHEN a user submits valid login credentials, THE Auth_System SHALL authenticate the user and establish a persistent session using Firebase Authentication.
2. WHEN an authenticated user navigates to the application, THE Router SHALL redirect the user to their role-specific dashboard.
3. IF a user submits invalid login credentials, THEN THE Auth_System SHALL display an error message indicating authentication failure.
4. WHEN a user clicks the logout button, THE Auth_System SHALL terminate the session and redirect the user to the login page.

### Requirement 3: Role-Based Route Protection

**User Story:** As a platform operator, I want routes protected by role, so that users can only access dashboards and features authorized for their role.

#### Acceptance Criteria

1. THE Router SHALL restrict access to admin routes exclusively to users with the admin role.
2. THE Router SHALL restrict access to startup routes exclusively to users with the startup role.
3. THE Router SHALL restrict access to mentor routes exclusively to users with the mentor role.
4. IF an unauthenticated user attempts to access a protected route, THEN THE Router SHALL redirect the user to the login page.
5. IF an authenticated user attempts to access a route outside their role, THEN THE Router SHALL redirect the user to their own role-specific dashboard.

### Requirement 4: Mentor Startup Discovery

**User Story:** As a mentor, I want to browse, search, and filter startup listings, so that I can discover startups aligned with my expertise.

#### Acceptance Criteria

1. WHEN a mentor navigates to the Mentor_Dashboard, THE Platform SHALL display a list of approved startups as Startup_Cards.
2. THE Startup_Card SHALL display the startup name, industry, startup stage, goals, funding stage, and description.
3. WHEN a mentor enters a search query, THE Platform SHALL filter the displayed startups by name or description matching the query.
4. WHEN a mentor selects filter criteria (industry, stage, or funding stage), THE Platform SHALL display only startups matching the selected filters.
5. WHEN a mentor applies multiple filters simultaneously, THE Platform SHALL display startups matching all selected filter criteria.

### Requirement 5: Mentor Interest Expression

**User Story:** As a mentor, I want to express interest in a startup, so that the startup knows I am available for mentorship.

#### Acceptance Criteria

1. THE Startup_Card SHALL display an "Interested" button for each startup listing.
2. WHEN a mentor clicks the "Interested" button on a Startup_Card, THE Matching_Service SHALL create an Interest_Record in Firestore linking the mentor to the startup with a timestamp.
3. WHEN a mentor has already expressed interest in a startup, THE Startup_Card SHALL display the "Interested" button in a disabled or "already interested" state.
4. IF the Interest_Record creation fails due to a network error, THEN THE Platform SHALL display an error message and allow the mentor to retry.

### Requirement 6: AI-Powered Mentor Recommendations for Startups

**User Story:** As a startup, I want to receive AI-generated mentor recommendations, so that I can find mentors best suited to my needs.

#### Acceptance Criteria

1. WHEN a startup navigates to the Startup_Dashboard, THE AI_Engine SHALL generate mentor recommendations using the Gemini API based on the startup profile, mentor profiles, and historical engagement data.
2. THE Mentor_Card in the AI suggestions section SHALL display the mentor name, expertise, industry specialization, Compatibility_Score, mentorship success rate, and AI reasoning summary.
3. THE Startup_Dashboard SHALL display AI-suggested mentors in the top section of the dashboard.
4. IF the Gemini API is unavailable, THEN THE AI_Engine SHALL fall back to Gemma for generating recommendations.
5. THE AI_Engine SHALL store each generated recommendation as a document in the ai_recommendations Firestore collection with the startup ID, mentor ID, score, reasoning, and timestamp.

### Requirement 7: Startup Mentor Decision Workflow

**User Story:** As a startup, I want to accept or reject mentors from both AI suggestions and interested mentors, so that I control which mentors I work with.

#### Acceptance Criteria

1. THE Startup_Dashboard SHALL display "Accept" and "Reject" action buttons on each Mentor_Card in both the AI suggestions section and the Mentors Interested section.
2. WHEN a startup clicks "Accept" on a Mentor_Card, THE Matching_Service SHALL create a Relationship_Record in Firestore linking the startup and mentor with status "active" and a timestamp.
3. WHEN a startup clicks "Reject" on a Mentor_Card, THE Matching_Service SHALL update the recommendation or interest record with status "rejected" and store the rejection in the engagement_history collection.
4. WHEN a startup accepts a mentor, THE Platform SHALL remove that mentor from the pending suggestions and interested lists.
5. THE AI_Engine SHALL use acceptance and rejection data from the engagement_history collection to improve future recommendations.

### Requirement 8: Mentors Interested Section on Startup Dashboard

**User Story:** As a startup, I want to see which mentors have expressed interest in me, so that I can evaluate and respond to potential mentors.

#### Acceptance Criteria

1. THE Startup_Dashboard SHALL display a "Mentors Interested" section below the AI suggestions section.
2. WHEN a mentor expresses interest in a startup, THE Startup_Dashboard SHALL display that mentor in the Mentors Interested section with their profile, expertise, experience, and availability.
3. THE Startup_Dashboard SHALL display the Mentors Interested section with the most recent interests shown first.
4. WHEN no mentors have expressed interest, THE Startup_Dashboard SHALL display an empty state message indicating no mentor interest yet.

### Requirement 9: AI Verification for Applications

**User Story:** As an admin, I want AI-generated verification summaries for startup and mentor applications, so that I can make informed approval decisions efficiently.

#### Acceptance Criteria

1. WHEN a startup or mentor submits an application with uploaded documents, THE Verification_Engine SHALL extract relevant information from the documents using the Gemini API.
2. THE Verification_Engine SHALL generate a verification summary including extracted company or mentor information, industry classification, and completeness assessment.
3. THE Verification_Engine SHALL produce a recommendation of "approve", "reject", or "pending review" for each application.
4. THE Admin_Dashboard SHALL display pending applications with the AI verification summary and recommendation.
5. WHEN an admin clicks "Approve" or "Reject" on an application, THE Platform SHALL update the user status in Firestore and record the admin decision with timestamp.
6. IF the Gemini API is unavailable during verification, THEN THE Verification_Engine SHALL fall back to Gemma for document analysis.

### Requirement 10: Document Upload and Management

**User Story:** As a startup, I want to upload and manage documents with visibility controls, so that I can share progress with mentors and admins while keeping sensitive documents private.

#### Acceptance Criteria

1. WHEN a startup uploads a document, THE Document_Service SHALL store the file in Firebase Storage and create a metadata document in the documents Firestore collection with file URL, upload timestamp, document type, and visibility setting.
2. THE Document_Service SHALL support document types including meeting minutes, monthly reports, and general documents.
3. WHEN a document is marked as "public", THE Platform SHALL make the document accessible to the associated startup, assigned mentors, and admins.
4. WHEN a document is marked as "private", THE Platform SHALL restrict document access to the startup that uploaded the document.
5. IF a file upload fails, THEN THE Document_Service SHALL display an error message and allow the user to retry the upload.
6. THE Document_Service SHALL validate uploaded files do not exceed 10 MB in size before initiating the upload.

### Requirement 11: Admin Ecosystem Analytics Dashboard

**User Story:** As an admin, I want to view ecosystem analytics and mentorship metrics, so that I can monitor platform health and make data-driven decisions.

#### Acceptance Criteria

1. THE Admin_Dashboard SHALL display ecosystem growth metrics including total startups, total mentors, and new registrations over time.
2. THE Admin_Dashboard SHALL display active mentorship count and mentorship engagement metrics.
3. THE Admin_Dashboard SHALL display analytics using charts and metric cards.
4. WHEN an admin navigates to the Admin_Dashboard, THE Platform SHALL retrieve current metrics from Firestore and display up-to-date analytics.

### Requirement 12: Firestore Data Model and Collection Linkages

**User Story:** As a developer, I want a well-structured Firestore schema with proper linkages, so that the platform can support matching workflows, behavioral learning, and analytics.

#### Acceptance Criteria

1. THE Firestore SHALL maintain collections for users, startups, mentors, mentor_interests, relationships, documents, feedback, ai_recommendations, and engagement_history.
2. THE users collection SHALL store user authentication ID, email, role, profile status, and creation timestamp.
3. THE mentor_interests collection SHALL store mentor ID, startup ID, timestamp, and status, with references linking to both the mentors and startups collections.
4. THE relationships collection SHALL store mentor ID, startup ID, status, creation timestamp, and engagement metrics, with references linking to both the mentors and startups collections.
5. THE engagement_history collection SHALL store user ID, action type (interested, accepted, rejected, viewed), target ID, and timestamp for behavioral learning.
6. THE ai_recommendations collection SHALL store startup ID, mentor ID, Compatibility_Score, reasoning, model used, and generation timestamp.

### Requirement 13: AI Behavioral Learning Data Collection

**User Story:** As a platform operator, I want all user interactions tracked for AI learning, so that recommendations improve over time.

#### Acceptance Criteria

1. WHEN a mentor clicks "Interested" on a startup, THE Platform SHALL record the interaction in the engagement_history collection.
2. WHEN a startup accepts or rejects a mentor, THE Platform SHALL record the decision in the engagement_history collection.
3. WHEN a startup views a mentor profile, THE Platform SHALL record the view event in the engagement_history collection.
4. THE AI_Engine SHALL query the engagement_history collection when generating recommendations to incorporate historical interaction patterns.

### Requirement 14: Platform UI and Layout

**User Story:** As a user, I want a clean, professional SaaS interface with intuitive navigation, so that I can use the platform efficiently.

#### Acceptance Criteria

1. THE Platform SHALL render exclusively in light mode with a clean, professional SaaS visual design.
2. THE Platform SHALL provide sidebar navigation for all authenticated views with role-appropriate menu items.
3. THE Platform SHALL be responsive and functional on desktop and tablet screen sizes.
4. THE Platform SHALL use shadcn/ui components styled with Tailwind CSS for consistent visual presentation.
5. THE Platform SHALL display analytics data using chart components and metric cards on dashboard views.
